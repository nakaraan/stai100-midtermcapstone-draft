Gradio vs. Streamlit
Two zero-config Python frameworks for chat interfaces
Gradio
— zero-config chat UI in ~5 lines
● gr.ChatInterface wraps a function — no HTML 
needed
● Built-in message history & streaming via yield
● Native file upload + multimodal (image, audio)
● Hugging Face Spaces — free hosting & share 
links
● Best for: ML demos, hackathons, rapid 
prototyping
Streamlit
— st.chat_message + st.chat_input primitives
● Streaming via st.write_stream + generators
● st.session_state holds full conversation 
memory
● Rich data viz (Plotly, Altair) in the same app
● Multi-page apps and sidebars out of the box
● Best for: data-heavy & analyst-facing 
dashboards
Both ship via pip and run locally in under 60 seconds — no frontend knowledge required.

Representational State 
Transfer (REST) is an 
architectural style for

systems. It treats every 
component as a resource 
accessible via standard 
URIs.

• Uses HTTP verbs: GET, 
POST, PUT, DELETE
• Stateless: Each request 
contains all info needed to 
process it
• Exchanges JSON or XML 
representations
 Why Use It?
• Scalability: Separates 
client and server concerns
• Interoperability: 
Language agnostic 
(Python, JS, Go)
• Simplicity: Leverages 
existing web infrastructure

Docker Containers vs. Virtual Machines
Docker Container
● Virtualization is at the application 
layer, runs on top of the host OS
● Manages dependencies in the 
application level
● Resource usage is less
● Can be started within seconds
● Easy to replicate, even with multiple 
instances
Virtual Machine
● Virtualization is both at the OS kernel 
and the application layer
● Manages dependencies at the system 
level
● Resource usage can be intensive
● Takes few minutes to start
● Difficult to replicate, especially with 
increasing number of VM instances

Docker Images vs. Containers
Image
● Blueprint of a container
● Created only once; contains 
dependencies, configs, scripts, and 
binaries
● Consists of multiple read-only layers
● Cannot attach volumes and 
networks
Container
● Runtime instance of an image
● Multiple instances can run from one 
image, each with unique state
● Features a single readable and 
writable layer
● Can attach volumes and networks

from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel
app = FastAPI()
QA: dict[str, str] = {} # Define default answers here
NO_ANSWER = "Sorry, I don't have an answer for that question."
class ChatRequest(BaseModel):
    message: str 
class ChatResponse(BaseModel):
    answer: str  
@app.get("/health")
def health():
    return {"status": "ok"} 
@app.post("/chat", response_model=ChatResponse)
def chat(req: ChatRequest):
    answer = QA.get(req.message, NO_ANSWER)
    return ChatResponse(answer=answer)

--- page 26 ---
Key Takeaways
What this dual-channel build teaches about LLM productionization
Separate channels, shared core
UI and API are thin wrappers over a single 
Python module. Refactor the core once; both 
channels improve together. Never duplicate 
prompt or RAG logic per channel.
Instrument from day one
Add structured logs and tracing before 
users arrive. Retrofitting observability into 
a running production system is 10× harder 
than instrumenting it at build time.
Docker is the contract
If it runs in your container, it runs on 
Render, Fly, Cloud Run, or a partner's VM. 
Pin Python, pin deps, expose ports — the 
rest is infrastructure detail.
Cost is a latency in disguise
Token usage and dollars per call are 
leading indicators of UX. A bloated prompt 
is slow AND expensive — monitoring 
catches both with the same dashboard.
